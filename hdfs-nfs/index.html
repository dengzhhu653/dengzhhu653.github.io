HDFS NFS

​ 在moosefs逐步退出后，提供的ceph只支持centos7以上版本，团队中的项目高度依赖hadoop生态，从底层存储(hdfs)，计算(mapreduce, tez, spark)到调度(yarn, azkaban)。 在一些团队合作的输出数据或者导出数据的场景中，需要大量的存储容量来存储临时数据。Hadoop HDFS内置了通过nfs的方式来挂载存储，用户可以使用普通的Linux文件操作命令来对挂载的HDFS中的文件进行操作，本文就是在操作部署HDFS NFS的一些总结和经验，让感兴趣的同学能够尽快熟悉和了解这种挂载方式。

  1. HDFS认证方式采用的IP受限的方式，不需要kerberos的配置；我们也不需要重启NameNode来增加用户代理配置，如下：

改成:

在配置nfs.exports.allowed.hosts项时，需要注意的是，多个实际相邻的IP不能配置成“192\.168\.0\.(11|22) ”或者“192.168.0.[11|22]”，而是应该配置成如下方式才能生效：

  1. 配置完成hdfs-site.xml文件之后，下一步则是要用root用户启动两个服务，同时设置HAOOP_USER_NAME变量：

[root]> $HADOOP_PREFIX/sbin/hadoop-daemon.sh –script $HADOOP_PREFIX/bin/hdfs start portmap
[root]$ $HADOOP_PREFIX/sbin/hadoop-daemon.sh –script $HADOOP_PREFIX/bin/hdfs start nfs3

不要用hdfs super用户来启动nfs3服务。如果服务启动中有问题，可以通过$HADOOP_LOG_DIR目录下的nfs3和portmap服务日志文件查看失败原因。

  1. 服务启动成功后，通过

showmount -e $nfs_server_ip

命令来确认HDFS是否可以被挂载，如果showmount命令没有找到，则需要安装nfs-utils插件：

sudo yum install nfs-utils

通过showmount命令，可以看到可以挂载的目录和一些机器地址信息，如：

Export list for c1-hd-dn13:
/user/yisou/hdfs-nfs 10.x.x.(44|45),10.x.x.(146|147),10.x.x.135
/tmp/dengzhihua 10.x.x.(44|45)

  1. 成功看到可以挂载信息后，下一步就是到目录的可挂载的IP上进行挂载操作，如：

[root]>mount -t nfs -o vers=3,proto=tcp,nolock c1-hd-dn13:/user/yisou/hdfs-nfs /hdfs

为了方便用户可以对挂载目录上的文件直接操作，proto去掉了sync配置，这样操作更友好。

在成功mount之后，可以查看一下/hdfs文件目录的用户组属性，可以发现出现一些奇怪的现象，用户属性可能是某个数字，也有可能是一个奇怪的用户，这是因为本地用户组和nfs3服务器上用户组id不一致导致的，解决这个问题可以通过配置用户id映射方式，可以通过在/etc/nfs.map(default)文件上配置：

uid 503 508 # Map the remote UID 503 the local UID 508
gid 503 508 # Map the remote GID 503 to the local GID 508
uid 508 508 # Map the remote UID 508 the local UID 508
uid 508 508 # Map the remote UID 508 the local UID 508

总结如下：

需要在hdfs-site.xml上配置，原nfs3只能挂载一个目录，经过我们的一些定制化后，可以挂载多个目录。

启动两个服务，

[root]> $HADOOP_PREFIX/sbin/hadoop-daemon.sh –script $HADOOP_PREFIX/bin/hdfs start portmap
[root]$ $HADOOP_PREFIX/sbin/hadoop-daemon.sh –script $HADOOP_PREFIX/bin/hdfs start nfs3

挂载客户端和nfs网关用户组信息不一致时，需要在/etc/nfs.map(default)配置一些用户id映射：

uid 503 508 # Map the remote UID 503 the local UID 508
gid 503 508 # Map the remote GID 503 to the local GID 508

经验教训
  • 通过NFS3的方式挂载hdfs，还有很多地方值得改进(这个需要深入去了解NFS传输协议，修改操作的返回结果)。比如，我在一个挂载的目录中删了一个不存在的文件，或者ls一个不存在文件，甚至连降一个存在其他硬盘中的文件重定向输出到目录中一个不存在的文件时，此时出现如：

​    [root@host aa]# ls aaa
​        ls: cannot access aaa: Invalid argument
​    [root@host aa]# ls
​        ls: cannot open directory .: Stale file handle
​     [root@host aa]#

出现这个问题Stale file handle后，随便一个文件操作命令，都没办法成功，除非重新执行挂载操作命令。原存在该目录中的文件不会丢失。

  • 这种方式只能适合一些低频次，读多于写的需求，比如一天一次以上等,不支持硬连接(hard link) 和随机写(random write)的场景。
